DataPilot

Mehdi Bargach · 2026

Un agent AI qui analyse n'importe quel fichier CSV : tu poses une question en langage naturel, et il génère du code d'analyse, l'exécute dans un environnement sécurisé, et te retourne la réponse avec un graphique — en 2 secondes.

1

Le problème

Situation : tu as un fichier CSV (export Shopify, Google Ads, Stripe, CRM...) et tu veux savoir quelque chose.

OptionCe qui se passePourquoi c'est cassé
Excel / Google Sheets Tableaux croisés, formules VLOOKUP, graphiques manuels 30-60 minutes pour répondre à UNE question. Chaque semaine. Et tu dois savoir ce que tu cherches.
ChatGPT Code Interpreter Upload le CSV, pose une question, obtiens une réponse La session expire après 30 min. Les fichiers disparaissent. Tu dois tout re-uploader. C'est un couteau suisse, pas un outil dédié.
Tableau / Power BI / Looker Dashboards interactifs, puissants Faits pour les équipes data. Un marketing manager ne va pas apprendre SQL pour savoir quel canal baisse.
Ne rien faire Prendre des décisions au feeling Personne ne cherche ce que tu n'as pas demandé. Les insights cachés restent cachés.

Le vrai besoin : poser une question en français ou en anglais à tes données, et recevoir la réponse avec un graphique — sans code, sans config, sans attente.

2

L'architecture — Prompt Chaining

DataPilot utilise un pattern appelé Prompt Chaining (chaînage de prompts). C'est le plus simple et le plus fiable des patterns d'agents AI :

PatternComment ça marchePourquoi DataPilot l'utilise
Prompt Chaining Des étapes fixées, dans un ordre précis. Chaque étape reçoit le résultat de la précédente. Fiable, prévisible, facile à débugger. Si une étape échoue, on sait laquelle.
ReAct (alternative) Le LLM décide lui-même quelles étapes prendre. Il réfléchit, agit, observe, et recommence. Plus flexible mais imprévisible. La latence peut varier de 2s à 60s selon le chemin choisi.
Pourquoi un seul appel LLM et pas deux ?

Le design initial prévoyait 2 appels : (1) générer le code, (2) résumer le résultat en langage naturel. Le 2ème appel ajoutait ~3 secondes sans apporter de valeur. Fix : le code généré inclut directement result = f"..." — la réponse est formatée dès l'exécution, pas besoin d'un deuxième appel.

Execution Python subprocess (pandas + matplotlib) Le code généré par le LLM est exécuté dans un processus isolé Pas de dépendance externe pour les calculs. Sécurisé (subprocess, pas eval).

Le flux : le frontend envoie une question + un CSV (ou un nom de dataset demo). Le backend orchestre 4 modules séquentiels (schema → codegen → sandbox → agent) et retourne la réponse + le chart en base64. 1 seul appel LLM par requête.

5

Le pipeline — 4 étapes

Chaque étape est un module Python indépendant, orchestré par agent.py.

SCHEMA
CSV → structure
→
CODEGEN
question → code
→
SANDBOX
code → résultat
→
ANSWER

1
SCHEMA CSV → structure · schema.py
Ce qui se passe

Le fichier CSV est lu avec détection automatique de l'encodage (UTF-8, Latin-1, etc.) et du séparateur (virgule, point-virgule, tabulation). Le module extrait les noms de colonnes, les types de données, et un échantillon des premières lignes.

Pourquoi on fait ça

Le LLM a besoin de comprendre la structure du CSV avant de pouvoir générer du code. C'est comme donner le sommaire d'un livre avant de poser des questions dessus — sans le sommaire, le LLM devinerait.

2
CODEGEN question → code pandas · codegen.py
Ce qui se passe

Le schéma du CSV + la question de l'utilisateur sont envoyés à GPT-4o. Le modèle retourne du code Python (pandas) qui répond à la question, incluant une variable result avec la réponse formatée.

Le modèle retourne aussi un flag needs_chart et optionnellement du code matplotlib pour générer un graphique.

Pourquoi on fait ça

Au lieu de demander au LLM de répondre directement (« le total est 2 millions »), on lui demande de générer du code. Le code est ensuite exécuté sur les vraies données. Résultat : les chiffres viennent d'un calcul réel, pas de la mémoire du LLM. Zéro hallucination sur les nombres.

Insight clé : le LLM ne voit jamais les données complètes. Il ne voit que le schéma (colonnes + types + échantillon). C'est le code généré qui touche les données. C'est pour ça qu'il ne peut pas halluciner les chiffres — ils viennent de l'exécution, pas de la génération.

Règle anti-hallucination dans le prompt

Le prompt contient cette règle : « If the question asks about data not present in the schema, say so explicitly. NEVER invent proxies. »

Exemple : « What is the customer satisfaction score? » sur un CSV sans colonne satisfaction → le modèle répond « This dataset does not contain customer satisfaction data » au lieu d'inventer un proxy (comme utiliser le profit).

3
SANDBOX code → résultat · sandbox.py
Ce qui se passe

Le code Python généré est exécuté dans un subprocess isolé. Le sandbox précharge le CSV en tant que DataFrame df, plus les librairies (pandas, numpy, matplotlib, seaborn). Il netttoie le code (supprime les imports, les print, etc.) avant de l'exécuter.

Pourquoi on fait ça

Sécurité : le code est exécuté dans un processus séparé, pas avec eval(). Si le code plante, le serveur principal n'est pas affecté.

Retry : si le code échoue, l'erreur est renvoyée au LLM qui génère une version corrigée. Maximum 2 tentatives.

4
ANSWER réponse + chart · agent.py
Ce qui se passe

L'orchestrateur (agent.py) récupère le result (texte) et le chart (PNG sauvé sur disque). Il assemble la réponse finale avec les timings et optionnellement des insights proactifs (anomalies, tendances détectées automatiquement).

Pourquoi on fait ça

Les insights proactifs répondent aux questions que l'utilisateur n'a pas posées. C'est la différence entre un outil qui attend tes questions et un analyste qui dit « hey, j'ai remarqué un truc bizarre dans tes données ».

6

La détection des méta-questions

Quand l'utilisateur demande « comment tu as calculé ça ? », l'agent ne doit pas recalculer — il doit expliquer sa méthode.

ApprocheFiabilitéRisque
Modifier le prompt ~70% Le LLM ignore l'instruction 1 fois sur 3. Pire : modifier le prompt cause des régressions sur toutes les autres questions.
Regex Python + prompt dédié ~100% Zéro impact sur le prompt principal. Détection déterministe (le code dit oui ou non, pas « peut-être »).

Principe général : quand une décision est binaire (oui/non), utiliser du code déterministe. Quand une décision est nuancée, utiliser le LLM. La détection de méta-questions est binaire. La génération de code analytique est nuancée. Chacune utilise l'outil adapté.

7

L'histoire du NO-GO — le micro-loop

Le premier round d'évaluation a donné un NO-GO à 55%. Le deuxième round, après 2 fixes, a donné un CONDITIONAL GO à 87.5%.

MétriqueRound 1 (GPT-4o-mini)Round 2 (GPT-4o)Delta
Score global55%87.5%+32.5pp
Hallucinations1 (BLOCKING)0Fixé
Multi-step40%100%+60pp
Adversarial17%83%+66pp
Latence3.5s2.1s-40%

Le bug critique : « What is the customer satisfaction score? » sur un CSV qui n'a pas de colonne satisfaction. Au lieu de refuser, le modèle a inventé que le profit pouvait servir de proxy pour la satisfaction. C'est une hallucination.

Les 2 fixes :

#FixImpact
1 Règle anti-hallucination dans le prompt : « If the data is not in the schema, say so. NEVER invent proxies. » Hallucination E15 → refus correct. 0/20 hallucinations.
2 Upgrade du modèle : GPT-4o-mini → GPT-4o Multi-step 40% → 100%. Le modèle génère du code correct pour les questions complexes (groupby + sort + head + format).

Insight clé : les micro-tests du BUILD (22/22 PASS) ne garantissent pas la qualité sur de nouvelles questions. L'éval avec un golden dataset NOUVEAU est indispensable — c'est là qu'on découvre les vrais problèmes. « In AI, shipping is easy. Evaluating is hard. »

Le problème : les équipes business passent 30 à 60 minutes dans Excel pour répondre à une seule question sur leurs données. ChatGPT peut aider mais la session expire après 30 minutes.

10

Concepts — si on te demande

ConceptC'est quoiExemple concret
Prompt Chaining Pattern d'agent AI où les étapes sont fixées et séquentielles. Chaque étape passe son résultat à la suivante. Plus fiable et prévisible que ReAct où le LLM décide du chemin. schema → codegen → sandbox → answer (toujours dans cet ordre)
Code generation Au lieu de demander au LLM de répondre directement, on lui demande de générer du code qui calcule la réponse. Les chiffres viennent d'un calcul réel, pas de la mémoire du LLM. « Total sales? » → result = f"Total: ${df['Sales'].sum():,.2f}"
Sandbox Environnement isolé pour exécuter du code généré par le LLM. Si le code plante ou fait quelque chose de dangereux, le serveur principal n'est pas affecté. Python subprocess avec timeout — pas eval()
Anti-hallucination Règle dans le prompt qui force le modèle à refuser au lieu d'inventer. « If the data is not in the schema, say so. NEVER invent proxies. » Les règles spécifiques marchent, les instructions vagues non. « Satisfaction score? » → « This dataset does not contain satisfaction data »
Golden dataset Jeu de 20 questions NOUVELLES (pas celles du développement) utilisées pour évaluer la qualité. Structurées par type : agrégation, tendance, comparaison, multi-étapes, adversarial, consistance. Les tests BUILD (22/22) ne trouvent pas les mêmes bugs que l'eval (55%)
Eval Gate Rituel de la Builder PM Method. 3 niveaux (BLOCKING / QUALITY / SIGNAL). 3 décisions (GO / CONDITIONAL GO / NO-GO). Un fail BLOCKING = retour obligatoire au BUILD. Round 1 : 1 hallucination BLOCKING → NO-GO → micro-loop → Round 2 : CONDITIONAL GO
Micro-loop Boucle BUILD ↔ EVALUATE. On fixe les problèmes trouvés par l'eval, puis on re-évalue. On boucle jusqu'à ce que les critères passent. 55% → prompt fix + model upgrade → 87.5%
Schema text Description textuelle de la structure du CSV : colonnes, types, échantillon. C'est ce que le LLM « voit » du fichier — jamais les données complètes. Column: Sales (float64), sample: [245.5, 731.2, 14.6]
Meta-question Question sur la méthode (« comment tu as calculé ça? »), pas sur les données. Détectée par regex Python pour éviter de modifier le prompt principal. Regex META_KEYWORDS → prompt dédié META_SYSTEM
FastAPI Framework Python pour créer des APIs. Tu définis des routes, FastAPI gère la validation, les erreurs, et la doc auto. @app.post("/analyze") crée la route principale
Render Plateforme cloud. Tu push du code sur GitHub, Render le détecte, l'installe et le met en ligne automatiquement. Push sur main → backend live en 2 minutes
Lovable Outil no-code qui génère du React + Tailwind via des prompts en langage naturel. « Ajoute un dataset selector avec 3 démos » → code React généré
