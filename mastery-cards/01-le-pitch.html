2

L'architecture — les 2 patterns d'agents AI

Quand on construit un agent AI (un programme qui exécute plusieurs étapes pour répondre à une demande), il y a deux grandes architectures possibles. Comprendre les deux est essentiel pour justifier le choix de l'une ou l'autre.

Pattern 1 — Chaînage d'étapes (Prompt Chaining)

L'analogie : une chaîne de montage en usine. Chaque poste fait un seul job et passe le résultat au poste suivant. L'ordre est fixé à l'avance, il ne change jamais.

Comment ça marche techniquement :

Le développeur définit une séquence d'étapes dans le code. Chaque étape reçoit le résultat de la précédente et produit un résultat pour la suivante. L'AI intervient à une ou plusieurs étapes précises (par exemple, générer du code), mais c'est le code du développeur qui orchestre le flux, pas l'AI.

DataPilot en chaînage — 4 étapes

schema.py (lire le fichier) → codegen.py (l'AI écrit le calcul — seul appel AI) → sandbox.py (exécuter le calcul) → agent.py (assembler la réponse). Toujours dans cet ordre. 2 secondes. Chaque étape est un module indépendant — si ça casse, on sait exactement où.

ForcesLimites

• Prévisible : même chemin, même durée, à chaque fois

• Débuggable : si ça rate, tu sais à quelle étape

• Testable : chaque module peut être testé isolément

• Coût maîtrisé : nombre fixe d'appels à l'AI

• Rigide : si la tâche demande un chemin différent, le pipeline ne peut pas s'adapter

• Pas d'exploration : l'AI ne peut pas décider « je vais vérifier quelque chose en plus »

Cas d'usage idéal : tâche répétitive avec un chemin connu à l'avance. Analyse de données, traitement de documents, pipelines RAG (question-réponse sur documents).

Pattern 2 — Agent autonome (ReAct)

L'analogie : un chef qui improvise. Il goûte, réfléchit, décide quoi faire ensuite. Parfois brillant, parfois 30 minutes pour un plat de 5 minutes.

Le nom technique : ReAct = Reason + Act (réfléchir + agir). Rien à voir avec React le framework front-end — même nom, concepts complètement différents.

Comment ça marche techniquement :

L'AI reçoit la question de l'utilisateur + une liste d'outils à sa disposition (lire un fichier, exécuter du code, appeler une API, chercher sur le web...). Ensuite, elle boucle toute seule en 3 temps :

ÉtapeCe qui se passeExemple concret
1. Reason (réfléchir) L'AI analyse la situation et décide quelle action prendre « Pour répondre à cette question sur les ventes, je dois d'abord comprendre la structure du fichier »
2. Act (agir) L'AI choisit un outil dans sa liste et l'utilise Appelle l'outil « lire le schéma du fichier »
3. Observe (observer) L'AI regarde le résultat et décide si elle a la réponse ou si elle doit faire un tour de plus « OK, je vois les colonnes. Maintenant je vais écrire un calcul pour les ventes par région » → retour à l'étape 1

Cette boucle Reason → Act → Observe se répète jusqu'à ce que l'AI décide qu'elle a assez d'information pour répondre. Le nombre de tours est variable — 3 tours pour une question simple, 10+ pour une question complexe.

ForcesLimites

• Flexible : s'adapte à n'importe quelle question, même imprévue

• Exploratoire : l'AI peut décider de vérifier un résultat, chercher plus loin, changer d'approche

• Multi-outils : peut combiner web, code, API, fichiers dans un seul flux

• Imprévisible : la latence peut varier de 2s à 60s+ selon le chemin choisi

• Coûteux : chaque tour = un appel AI supplémentaire (coût et temps)

• Dur à débugger : il faut relire tout le « raisonnement » pour comprendre où ça a déraillé

• Risque de boucle : l'AI peut tourner en rond si elle ne trouve pas la réponse

Cas d'usage idéal : tâche exploratoire avec un chemin inconnu à l'avance. Recherche web multi-sources, audit de site (crawler + analyser + synthétiser), assistant de code qui doit comprendre un projet entier.

Pourquoi le chaînage pour DataPilot

Analyser des données suit toujours le même chemin : comprendre le fichier → écrire le calcul → exécuter → répondre. Le chemin est connu à l'avance, la tâche est répétitive. Laisser l'AI décider du chemin = complexité et imprévisibilité pour zéro gain.

Preuve par le négatif : LeakFinder, un autre side project, utilisait un agent ReAct en 9 étapes pour auditer des sites web. La latence variait de 2 secondes à 22 minutes selon les sites. L'imprévisibilité a tué le projet — NO-GO pour risque démo trop élevé.

Règle de décision : si tu peux dessiner le chemin à l'avance → chaînage. Si chaque question peut nécessiter un chemin différent → agent autonome. La question n'est pas « quel pattern est meilleur » mais « quel pattern correspond à ta tâche ».

Le flux : l'interface envoie une question + un fichier de données. Le serveur orchestre 4 étapes dans l'ordre (structure → calcul → exécution → réponse) et retourne le résultat + le graphique. 1 seul appel à l'AI par question.

Le pipeline — 4 étapes

Le pipeline fonctionne comme une chaîne de montage : chaque étape fait un seul job, puis passe le résultat à la suivante. 4 étapes, toujours dans le même ordre.

SCHEMA
CSV → structure
→
CODEGEN

→
SANDBOX
code → résultat
→
ANSWER
réponse + chart
1
SCHEMA CSV → structure · schema.py
Ce qui se passe

Le fichier CSV est lu avec détection automatique de l'encodage (UTF-8, Latin-1, etc.) et du séparateur (virgule, point-virgule, tabulation). Le module extrait les noms de colonnes, les types de données, et un échantillon des premières lignes.

Pourquoi on fait ça

L'AI a besoin de comprendre la structure du fichier avant de pouvoir écrire le calcul. C'est comme donner le sommaire d'un livre avant de poser des questions dessus — sans le sommaire, l'AI devinerait.

2
CODEGEN question → instructions de calcul
Ce qui se passe

La structure du fichier + la question de l'utilisateur sont envoyées à GPT-4o. Le modèle écrit des instructions de calcul (du code) qui répondent à la question. Il décide aussi si un graphique est pertinent, et si oui, écrit les instructions pour le générer.

Pourquoi on fait ça

C'est l'astuce clé du projet. Au lieu de demander à l'AI de répondre directement (« le total est 2 millions »), on lui demande d'écrire la formule de calcul. Cette formule est ensuite exécutée sur les vraies données. Résultat : les chiffres viennent d'un calcul réel, pas de la mémoire de l'AI. Zéro hallucination sur les nombres.

Insight clé : l'AI ne voit jamais les données complètes. Elle ne voit que le sommaire du fichier (noms des colonnes, types, quelques exemples). C'est le calcul généré qui touche les vraies données. C'est pour ça que les chiffres ne peuvent pas être inventés — ils viennent d'un calcul réel, pas de la mémoire de l'AI.

Règle anti-hallucination

Les instructions données à l'AI contiennent cette règle précise : « Si la donnée demandée n'existe pas dans le fichier, dis-le clairement. N'invente JAMAIS de substitut. »

Exemple : « Score de satisfaction client ? » sur un fichier sans colonne satisfaction → l'AI répond « Ce fichier ne contient pas de donnée sur la satisfaction » au lieu d'inventer un substitut (comme utiliser le profit comme approximation).

3
SANDBOX instructions → résultat
Ce qui se passe

Les instructions de calcul générées à l'étape 2 sont exécutées dans un espace isolé (un « sandbox » — comme une salle d'examen où le code tourne seul, séparé du reste du système). Le fichier de données est chargé dans cet espace, le calcul tourne, et le résultat sort.

Pourquoi on fait ça

Sécurité : le code tourne dans un processus séparé. Si le calcul plante ou fait quelque chose d'inattendu, le serveur principal n'est pas touché.

Auto-correction : si le calcul échoue, l'erreur est renvoyée à l'AI qui génère une version corrigée. Maximum 2 tentatives.

4
ANSWER réponse + chart · agent.py
Ce qui se passe

L'orchestrateur (agent.py) récupère le result (texte) et le chart (PNG sauvé sur disque). Il assemble la réponse finale avec les timings et optionnellement des insights proactifs (anomalies, tendances détectées automatiquement).

Pourquoi on fait ça

Les insights proactifs répondent aux questions que l'utilisateur n'a pas posées. C'est la différence entre un outil qui attend tes questions et un analyste qui dit « hey, j'ai remarqué un truc bizarre dans tes données ».

6

La détection des méta-questions

Quand l'utilisateur demande « comment tu as calculé ça ? », l'agent ne doit pas recalculer — il doit expliquer sa méthode.

ApprocheFiabilitéRisque
Demander à l'AI de détecter ~70% L'AI ignore l'instruction 1 fois sur 3. Pire : ajouter cette instruction dans le prompt principal cause des effets de bord sur toutes les autres questions.
Détection par le code ~100% Le code vérifie si la question contient des mots-clés comme « comment », « pourquoi », « explain ». C'est binaire (oui ou non), pas « peut-être ». Zéro impact sur le reste du système.

Principe général : quand une décision est oui/non, le code classique est plus fiable que l'AI. Quand une décision demande de la compréhension (comprendre une question, écrire un calcul), l'AI est meilleure. DataPilot utilise chaque outil là où il excelle : le code pour le tri (méta-question ou pas), l'AI pour l'analyse.

L'histoire du NO-GO — le micro-loop

Le premier round d'évaluation a donné un NO-GO à 55%. Le deuxième round, après 2 fixes, a donné un CONDITIONAL GO à 87.5%.

Le bug critique : on demande « Score de satisfaction client ? » sur un fichier qui n'a pas de colonne satisfaction. Au lieu de refuser, l'AI a inventé que le profit pouvait servir d'approximation pour la satisfaction. C'est une hallucination — une réponse qui a l'air correcte mais qui est fabriquée.

Les 2 corrections :

#CorrectionImpact
1 Règle anti-hallucination dans les instructions de l'AI : « Si la donnée n'existe pas, dis-le. N'invente jamais de substitut. » L'hallucination disparait. 0/20 données inventées.
2 Passage au modèle plus puissant : GPT-4o-mini → GPT-4o Questions complexes (tri + regroupement + classement) : 40% → 100%. Le modèle plus puissant écrit des calculs corrects même pour les questions en plusieurs étapes.

Insight clé : les tests faits pendant le développement (22/22 réussis) ne garantissent pas la qualité sur de nouvelles questions. L'évaluation avec des questions que le système n'a jamais vues est indispensable — c'est là qu'on découvre les vrais problèmes. « En AI, livrer c'est facile. Évaluer, c'est dur. »

8

L'évaluation — Eval Gate

L'Eval Gate évalue le produit sur 6 critères, organisés en entonnoir — du plus critique au moins critique. On vérifie d'abord que l'outil ne ment pas, puis qu'il calcule juste, puis qu'il répond correctement dans l'ensemble.

CoucheCritèresQuestionNiveau
1. Confiance G1 + G2 Est-ce que l'outil invente des données ? Est-ce que les calculs simples sont justes ? BLOCKING
2. Qualité G3 + G4 Est-ce que la précision globale est suffisante ? Les graphiques sont-ils corrects ? QUALITY
3. Performance G5 Est-ce que ça répond assez vite ? SIGNAL

Pourquoi ces niveaux et ces seuils

CritèreNiveauSeuilJustification
G1 — Zéro hallucination BLOCKING 0 donnée inventée sur 20 questions Un outil d'analyse qui invente des chiffres est pire qu'inutile — il donne une fausse confiance. Si l'AI ne sait pas, elle doit le dire. Non-négociable.
G2 — Calculs simples exacts BLOCKING 100% sur les 4 questions de calcul simple (sommes, moyennes, %) Ce sont les questions les plus basiques (« quel est le total des ventes ? »). Si l'outil se trompe là-dessus, c'est un problème fondamental — autant utiliser Excel.
G3 — Précision globale QUALITY ≥ 90% (18/20 questions) Inclut les questions complexes (classements, comparaisons multi-étapes, questions pièges). Le seuil à 90% laisse une marge de 2 erreurs sur 20. En dessous, le produit n'est pas assez fiable pour être montré.
G4 — Graphiques corrects QUALITY 100% des graphiques générés correspondent aux données Un graphique faux est plus dangereux qu'un chiffre faux — on le lit en 1 seconde sans vérifier. Les graphiques générés doivent être irréprochables, même si le taux de génération est faible.
G5 — Latence SIGNAL < 15s médiane Produit démo sur Render à $7/mois, pas un SaaS en production. Le seuil est généreux volontairement. En production, ce serait QUALITY avec un seuil à < 5s.
Règles de décision

GO = 0 BLOCKING fail + 0 QUALITY fail → on met en ligne.
CONDITIONAL GO = 0 BLOCKING fail + ≥1 QUALITY/SIGNAL fail → on met en ligne avec conditions documentées.
NO-GO = ≥1 BLOCKING fail → retour obligatoire au développement (micro-loop).

Méthodologie

L'évaluation est faite sur un jeu de 20 questions nouvelles (aucune réutilisée du développement), structurées par type : calcul simple (4), tendance (3), comparaison (4), multi-étapes (5), questions pièges (3), consistance (1). Chaque réponse est notée : 1.0 = correct, 0.5 = partiellement correct, 0.0 = incorrect, -0.5 = hallucination. La moyenne donne le score final.

4 conditions documentées

#ConditionDétailImpact
1 Calcul de marge « Quelle catégorie a la pire marge de profit ? » — l'AI calcule la moyenne des marges par ligne (3.88%) au lieu du profit total / ventes totales (2.5%). Deux méthodes possibles, l'AI choisit la mauvaise pour ce contexte. Accepté. Fixable par une règle dans les instructions : « marge = total profit / total ventes ».
2 Coût par conversion « Quel canal a le coût par conversion le plus bas ? » — l'AI confond deux colonnes (coût par clic vs coût par conversion), donne 0.50$ au lieu de 2.02$. Accepté. Fixable en enrichissant la description des colonnes dans le sommaire du fichier.
3 Taux de génération de graphiques Seulement 2 graphiques générés sur 8 questions qui le méritaient (25%). Les graphiques produits sont corrects, mais l'AI ne les produit pas assez souvent. Accepté. Fixable par une règle de génération plus agressive dans les instructions.
4 Refus implicite sur les dates « Dépenses marketing en 2023 ? » — l'AI retourne « 0.00$ » au lieu de dire clairement « pas de données pour 2023 ». Pas faux, mais trompeur. Accepté. Fixable par une règle : « si les dates demandées ne sont pas dans le fichier, dis-le ».

Décision : CONDITIONAL GO → mise en ligne. 0 BLOCKING fail, 1 QUALITY fail (G3 : 87.5% < 90% — écart d'1 question). Les 4 conditions sont des optimisations d'instructions, pas des bugs critiques. Le produit est fiable à 87.5% sans aucune hallucination.

Lovable Outil no-code qui génère du React + Tailwind via des prompts en langage naturel. « Ajoute un dataset selector avec 3 démos » → code React généré

Réponses aux questions test

Référence — à lire après avoir tenté de répondre seul (Section 11).

1. Quel problème DataPilot résout qu'Excel et ChatGPT ne résolvent pas ?

Excel : tu dois savoir ce que tu cherches ET savoir le formuler (VLOOKUP, tableaux croisés). 30 à 60 minutes pour UNE question. Pas d'insights automatiques — tu ne trouves que ce que tu cherches.

ChatGPT : c'est un couteau suisse (texte, image, code, données). Il n'est pas optimisé pour l'analyse de données : session qui expire, fichiers qui disparaissent, pas d'insights proactifs, pas de graphiques adaptés automatiquement.

DataPilot : outil dédié à l'analyse. Tu poses ta question, tu reçois la réponse avec un graphique en 2 secondes. Et il te signale des anomalies que tu n'as pas demandées.

2. C'est quoi le chaînage d'étapes vs l'agent autonome ? Pourquoi ce choix ?

Le chaînage (Prompt Chaining) : le développeur définit des étapes dans le code, dans un ordre fixe. L'AI intervient à un point précis, mais c'est le code qui orchestre le flux. Résultat : prévisible, débuggable, coût fixe.

L'agent autonome (ReAct = Reason + Act) : l'AI reçoit des outils et décide toute seule quoi faire. Elle boucle : réfléchir → choisir un outil → observer → recommencer, jusqu'à ce qu'elle ait la réponse. Le nombre de tours est variable (3 à 15+). Résultat : flexible mais imprévisible, coûteux, dur à débugger.

Pourquoi le chaînage pour DataPilot : l'analyse de données suit toujours le même chemin (comprendre le fichier → écrire le calcul → exécuter → répondre). Le chemin est connu, la tâche est répétitive. Laisser l'AI décider = complexité et imprévisibilité pour zéro gain.

Preuve par le négatif : LeakFinder, un autre side project, utilisait un agent ReAct en 9 étapes pour auditer des sites web. Latence : 2s à 22 minutes. L'imprévisibilité a tué le projet (NO-GO).

Règle de décision : si tu peux dessiner le chemin à l'avance → chaînage. Si chaque question peut nécessiter un chemin différent → agent autonome.

3. Pourquoi un seul appel à l'AI par question ?

Le design initial prévoyait 2 appels : (1) l'AI écrit les instructions de calcul, (2) une deuxième AI résume le résultat en langage naturel.

Le problème : le 2ème appel ajoutait ~3 secondes sans apporter de valeur. La solution : on demande à l'AI d'inclure directement la réponse formatée dans les instructions de calcul. Le calcul produit à la fois le chiffre ET la phrase de réponse.

Résultat : même qualité, moitié du temps, moitié du coût.

4. Le pipeline en 4 étapes

Étape 1 — SCHEMA : le fichier de données est lu, et on en extrait un sommaire (noms des colonnes, types, quelques exemples). C'est ce sommaire qui sera envoyé à l'AI — pas les données complètes.

Étape 2 — CODEGEN : le sommaire + la question de l'utilisateur sont envoyés à GPT-4o. L'AI écrit les instructions de calcul (du code) qui répondent à la question.

Étape 3 — SANDBOX : les instructions sont exécutées dans un espace isolé avec les vraies données. Si le calcul plante, on renvoie l'erreur à l'AI pour qu'elle corrige (max 2 tentatives).

Étape 4 — ANSWER : le résultat du calcul (texte + graphique si pertinent) est assemblé et renvoyé à l'utilisateur, avec optionnellement des insights proactifs.

5. Pourquoi l'AI ne voit jamais les données complètes ?

L'AI ne reçoit que le sommaire du fichier (colonnes, types, quelques exemples) — jamais les milliers de lignes de données.

Avantage 1 : les chiffres ne peuvent pas être inventés. L'AI écrit la formule de calcul, mais ce sont les vraies données qui produisent le chiffre. Si l'AI se trompe dans la formule, le résultat sera faux — mais il ne sera pas inventé.

Avantage 2 : rapidité et coût. Envoyer un sommaire de 20 lignes à l'AI coûte moins cher et va plus vite qu'envoyer 10 000 lignes de données.

6. C'est quoi une hallucination ? L'exemple E15.

Une hallucination = quand l'AI génère une réponse qui a l'air correcte mais qui est fabriquée. Elle n'a pas fait d'erreur de calcul — elle a inventé une donnée qui n'existe pas.

Exemple E15 : on demande « Score de satisfaction client ? » sur un fichier qui n'a pas de colonne satisfaction. Au lieu de refuser, l'AI a décidé que le profit pouvait servir d'approximation pour la satisfaction. C'est une réponse structurée, argumentée — et complètement inventée.

Pourquoi c'est grave : l'utilisateur fait confiance au chiffre. Il ne sait pas que l'AI a inventé le lien profit→satisfaction. C'est pour ça que c'est classé BLOCKING — un outil d'analyse qui invente des données est pire qu'inutile.

7. Comment la règle anti-hallucination fonctionne ?

On ajoute dans les instructions de l'AI une règle précise et spécifique : « Si la donnée demandée n'existe pas dans le fichier, dis-le clairement. N'invente jamais de substitut. »

Pourquoi les instructions vagues ne marchent pas : « sois honnête » ou « ne mens pas » ne fonctionne pas — l'AI ne croit pas mentir quand elle invente un proxy. Elle essaie d'aider. Il faut une règle spécifique qui couvre le cas exact : donnée absente → refus, pas approximation.

Résultat : après la règle, 0/20 hallucinations. L'AI refuse correctement chaque fois que la donnée n'existe pas.

8. L'histoire du NO-GO en 30 secondes

Round 1 (modèle léger, GPT-4o-mini) : 55% de précision, 1 hallucination. Décision : NO-GO — retour au développement.

2 corrections : (1) règle anti-hallucination dans les instructions, (2) passage au modèle plus puissant (GPT-4o).

Round 2 : 87.5%, zéro hallucination, latence en baisse (-40%). Décision : CONDITIONAL GO — on met en ligne avec 4 conditions documentées.

Le delta : +32.5 points de pourcentage. Les questions complexes (multi-étapes) : 40% → 100%. Les questions pièges : 17% → 83%.

9. Pourquoi les 22/22 tests du développement n'ont pas prédit le 55% ?

Les tests du développement vérifient que le code fonctionne : est-ce que le pipeline tourne ? Est-ce que les modules se connectent ? Est-ce que le résultat a le bon format ?

L'évaluation vérifie que le produit répond bien sur des questions nouvelles — des questions que le système n'a jamais vues pendant le développement.

C'est la différence entre réussir les exercices du cours (22/22) et passer l'examen final (55%). Les vrais bugs — hallucination, erreurs sur questions complexes, refus manquants — n'apparaissent que sur du terrain inconnu.

10. Code déterministe > AI probabiliste ? L'exemple des méta-questions.

Le problème : quand l'utilisateur demande « comment tu as calculé ça ? », c'est une question sur la méthode, pas sur les données. Il ne faut pas relancer un calcul.

Option 1 — demander à l'AI de détecter : on ajoute dans les instructions « si c'est une question sur la méthode, explique au lieu de calculer ». Résultat : ~70% de fiabilité. L'AI ignore l'instruction 1 fois sur 3, et ça crée des effets de bord sur les autres questions.

Option 2 — le code détecte : avant d'envoyer la question à l'AI, le code vérifie si elle contient des mots comme « comment », « pourquoi », « explain ». Si oui → redirection vers un chemin dédié. C'est binaire (oui ou non), ~100% fiable, zéro impact sur le reste.

Principe : quand une décision est oui/non, le code classique bat l'AI. Quand une décision demande de la compréhension, l'AI bat le code.

11. Le schéma d'architecture

Interface (Lovable / React + Tailwind) → envoie la question + le fichier via HTTPS → Serveur (FastAPI sur Render) qui orchestre 4 modules dans l'ordre :

1. schema.py : lit le fichier, extrait le sommaire (colonnes, types, exemples)

2. codegen.py : envoie le sommaire + la question à GPT-4o → reçoit les instructions de calcul (seul appel AI)

3. sandbox.py : exécute les instructions dans un espace isolé → résultat + graphique

4. agent.py : assemble la réponse + insights proactifs → retourne le tout à l'interface en JSON

12. L'Eval Gate : 3 niveaux, 3 décisions

3 niveaux de sévérité :

• BLOCKING = si ça échoue, on ne met pas en ligne. Non-négociable. Pour DataPilot : zéro hallucination + calculs simples exacts.

• QUALITY = seuil configurable. On évalue si c'est acceptable. Pour DataPilot : précision globale ≥ 90% + graphiques corrects.

• SIGNAL = on mesure pour observer. Ne bloque pas. Pour DataPilot : latence < 15s.

3 décisions possibles :

• GO = 0 BLOCKING fail + 0 QUALITY fail → on met en ligne.

• CONDITIONAL GO = 0 BLOCKING fail + ≥1 QUALITY fail → on met en ligne avec conditions documentées.

• NO-GO = ≥1 BLOCKING fail → retour obligatoire au développement.

DataPilot : Round 1 = NO-GO (1 hallucination BLOCKING). Round 2 = CONDITIONAL GO (0 BLOCKING, 1 QUALITY fail à 87.5% au lieu de 90%).

13. Pourquoi GPT-4o et pas GPT-4o-mini ?

GPT-4o-mini a été testé en premier (moins cher, plus rapide). Résultat : 55% de précision, 1 hallucination, et seulement 40% sur les questions complexes.

Après passage à GPT-4o : 87.5%, 0 hallucination, 100% sur les questions complexes. Et paradoxalement plus rapide (2.1s vs 3.5s de médiane) — le modèle plus puissant écrit des calculs corrects du premier coup, sans avoir besoin de retenter.

Trade-off : GPT-4o coûte ~10x plus par requête. Acceptable pour un produit démo. En production à grande échelle, il faudrait optimiser les coûts. Mais ici, la qualité est décisive — un outil d'analyse qui se trompe 1 fois sur 2 n'a pas de valeur.

14. C'est quoi un sandbox ? Pourquoi c'est important ?

Un sandbox = un espace isolé où le code tourne seul, séparé du reste du système. Comme une salle d'examen : ce qui se passe dedans ne peut pas affecter l'extérieur.

Pourquoi c'est critique : l'AI génère du code à exécuter. Ce code pourrait planter, boucler infiniment, ou tenter des opérations dangereuses. Le sandbox garantit que même si le code généré fait n'importe quoi, le serveur principal n'est pas touché.

En pratique : le calcul tourne dans un processus séparé avec un temps limite. S'il dépasse le temps ou plante, il est coupé proprement. Le serveur reste en ligne pour les autres utilisateurs.

15. « En AI, livrer c'est facile. Évaluer, c'est dur. »

Livrer : DataPilot a été développé en vertical slices (squelette → fonctionnalités). 22 tests de développement, tous réussis. Le pipeline fonctionne, le code est propre, le serveur tourne. Tout a l'air parfait.

Évaluer : on pose 20 questions nouvelles. Résultat : 55%. Le modèle hallucine, se trompe sur les questions complexes, ne refuse pas quand il devrait. Les vrais problèmes n'apparaissent que quand on teste avec des questions inconnues.

La leçon : en AI, le fait que le code fonctionne ne dit rien sur la qualité des réponses. Un pipeline qui tourne à 100% peut produire des réponses fausses à 45%. L'évaluation sur des questions nouvelles est l'étape la plus importante — et celle que la plupart des gens sautent.
